# Commit-Reveal & Vote Privacy

Vote privacy is critical for honest dispute resolution. If jurors can see how others vote, they may be influenced by social pressure or bribery. Kleros V2 uses commit-reveal voting with planned Shutter Network integration for enhanced privacy.

## Commit-Reveal Mechanism

Courts with `hiddenVotes: true` use a two-phase voting process:

### Commit Phase
Jurors submit a hash of their vote:
```
commitment = keccak256(abi.encodePacked(choice, salt))
```
- `choice` is the ruling option (1, 2, etc.)
- `salt` is a random value known only to the juror

### Reveal Phase
Jurors reveal their vote by providing the original `choice` and `salt`. The contract verifies:
```
keccak256(abi.encodePacked(choice, salt)) == commitment
```

If a juror fails to reveal, they are treated as incoherent and lose stake.

## Shutter Network Integration

<Info>
Shutter Network integration is under development for enhanced vote encryption.
</Info>

Shutter Network provides **threshold encryption** — votes are encrypted such that no single party can decrypt them. Only after the voting period ends does the decryption key become available, ensuring true privacy during the voting period.

This improves on basic commit-reveal by:
- Preventing jurors from selectively revealing their votes to others before the reveal phase
- Eliminating the need for jurors to remember and manage salts
- Providing cryptographic guarantees rather than game-theoretic ones

## Juror Fraud Protection

V2 introduces mechanisms to detect and penalize:
- **Pre-reveal fraud**: If a juror's vote is suspected of being pre-revealed, it can be challenged in a separate dispute
- **Cartelling**: Evidence of juror coordination (e.g., on-chain transactions between juror addresses, screenshot of discussions) can be submitted as a challenge
- These challenges don't interfere with the main dispute
